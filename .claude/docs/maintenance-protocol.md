# 制度檔維護協議

> 管的是 `.claude/docs/*.md` + `CLAUDE.md` 這套制度本身怎麼演化。
> 原則：制度要活（踩坑教訓寫得回來），但不能被慢慢架空或無限膨脹。
> last-updated: 2026-07-04（Fable 5 建立）

---

## 1. 改之前

- **一律先備份**：`cp <檔> .claude/docs/backups/<檔名>.$(date +%Y-%m-%d).bak`。backups/ 每個檔只留最近 2 份，多的刪掉。
- 更新檔頭的 `last-updated` 行（日期 + 一句話說明改了什麼）。

## 2. 權限分級

**可以自行改（不用問使用者）**：
- 在「教訓紀錄」節新增條目（格式見 §3）。
- 修正**事實錯誤**：路徑不存在、指令打錯、工具名 / schema 已變、資源 ID 過期。修正時註明驗證方式（例如「2026-08-01 實測 Agent tool 已無 fable 選項」）。
- delegation-templates.md 增加新任務型態的模板、或在既有模板補充實例。
- 精簡：合併重複條目、刪已標註過期的內容（見 §4）。

**先問使用者才能改**：
- 判準門檻（升降級規則、重試上限、完成階梯定義、§3 停下來問的清單）——這些是制度的骨架，改了會影響所有後續 session 的行為。
- 刪除任何規則（精簡合併不算刪除；「這條我覺得沒必要」算）。
- CLAUDE.md 的路由表結構、Session 紀律三條。
- 新增「例外」：如果你想在某任務中違反某條規則，可以（規則是判準不是法律），但**必須在回報中明說違反了哪條、為什麼**；若同類例外出現第二次，就該提議修改規則本身（問使用者），而不是繼續累積默許的例外。

## 3. 踩坑教訓怎麼寫回來

每次踩坑（自己的或 subagent 的），教訓寫到**對應檔案**的「教訓紀錄」節：
- 調度/委派/驗收出的問題 → `model-dispatch.md` §8
- 判斷失誤（誤判完成、該問沒問、方向錯了沒換路）→ `judgment-rubrics.md` 對應章節下加正/反例
- 專案知識類（架構、部署、第三方服務的坑）→ 照舊寫 memory 檔（這不歸本協議管）

**格式（一條一行到三行）**：
```
- [YYYY-MM-DD] 症狀：{發生什麼} → 根因：{為什麼} → 規則：{一句可執行的話}
```
範例：
```
- [2026-07-04] 症狀：照 memory 找 feat/onboarding-tour 分支找不到 → 根因：memory 過期未更新 →
  規則：引用 memory 的分支前先 git branch -a | grep 驗證（已納入 rubrics §6）。
```

## 4. 膨脹控制（觸發即執行）

- 單一制度檔 **>300 行**、或教訓紀錄 **>15 條** → 做一次精簡：同類合併、已內化成正式規則的教訓條目刪除（規則本體留下）、一次性的刪掉。
- CLAUDE.md **>150 行** → 檢查是否有內容該抽到引用檔。CLAUDE.md 每 session 都載入，是最貴的地產。
- 精簡屬於「可自行改」，但精簡後在回報裡告知使用者刪了什麼類別。

## 5. 制度與現實衝突時

文件說 X、實際環境是 Y（工具 schema 變了、模型名單變了、專案結構改了）：
1. **以現實為準**行動（不要為了遵守文件而用壞掉的方式做事）。
2. 當場修正文件（屬「事實錯誤」，可自行改），註明驗證方式。
3. 若衝突的是判準類規則（不是事實），先照 §2 問使用者。

## 6. 每季健檢（使用者提到「檢查制度」或距 last-updated 超過 ~3 個月時建議執行）

派一個 fresh-context agent 讀全部制度檔，回報：
- 互相矛盾的規則、指向不存在路徑/工具的引用、從沒被用過的規則（問使用者要不要刪）。
- MEMORY.md 是否又長出 inline 內容（應為純 index）。
- 用 delegation-templates.md 的審查模板派工。
