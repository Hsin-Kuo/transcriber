# 判斷力 Rubrics（弱模型可執行版）

> 這些判準把「資深工程師的直覺」翻成可機械執行的規則。每條附正例與反例。
> 被 CLAUDE.md、model-dispatch.md 引用；章節編號不要隨意變動（§N 被外部引用）。
> last-updated: 2026-07-04（Fable 5 建立）

---

## §1 何時該升級模型（或升級 effort）

**判準**：滿足任一 → 按 model-dispatch.md §6 升級：
- 同一子任務已失敗（不符驗收條件）達到升級門檻（haiku 一次 / sonnet 兩次）。
- 任務需要「跨 3 個以上模組的因果推理」或「在多個彼此衝突的目標間取捨」。
- 涉及 auth / payment / IAM / 資料遷移 / 不可逆操作的**設計決策**（執行可以低階，決策要高階）。

**正例**：sonnet 兩次都沒修好「SSE 偶發斷線」，第二次還把 heartbeat 改壞 → 帶兩次失敗軌跡升 opus 重新診斷。
**反例（不該升級）**：haiku 批次改檔時漏了 1 個檔——這是規格沒列全，補上檔案清單重派 haiku 即可，不是能力問題。

## §2 何時算真的完成（完成階梯）

**階梯**：`①設計定案 → ②實作完 → ③單元測試過 → ④實機 smoke 過 → ⑤commit → ⑥PR 開了 → ⑦merged → ⑧deployed`

**判準**：
- 回報時**必須標註目前在第幾階**。「完成」二字只能用在「使用者要求的那一階已到 + 該階驗證已跑過」。
- 各階的驗證：③要貼測試指令與結果摘要；④要用 `/verify` 或手動實跑該流程並描述觀察到的行為；⑧要打 prod/staging 實際確認。
- 本專案的預設期望：使用者說「做 X」通常指到 ⑥（開 PR）；merge 由使用者決定（見 memory：別自動輪詢 PR）。

**正例**：「修好了 editor 的 Option+Delete（第④階：本地實跑確認、320 測試全過），已開 PR #230，等你 merge。」
**反例**：「已完成標點強化的修正」——但測試沒跑、只是改完程式碼。這是第②階，不是完成。

## §3 何時該停下來問使用者

**判準**：滿足任一 → 停下來問，其餘情況自己決定並在回報中說明假設：
- **不可逆或外部可見**：刪除資料、動 prod、發 email、對外 API 呼叫有副作用、花錢（開 EC2、買服務）。
- **規格分岔且兩邊代價都高**：兩種合理解讀，選錯要重做一半以上。
- **要推翻使用者先前的明確決策**（例如 merge policy、tier 設計）——先報告矛盾，不要自行改。
- **品味 / 產品判斷**：UX 文案、定價、功能取捨。給 2-3 個選項 + trade-off，讓使用者拍板。

**正例**：發現修 bug 最乾淨的方法要改 DB schema（影響 prod 既有資料）→ 停下：「有 A（workaround，今天能上）和 B（schema migration，乾淨但要遷移），建議 B，你選？」
**反例（不該問）**：「變數名要叫 `taskId` 還是 `task_id`？」——照該檔案既有慣例，自己決定。

## §4 方向錯了的訊號（換路，不是重試）

**判準**：出現任一訊號 → 停止當前路線。換路 = 換方法 / 升級模型 / 問使用者，禁止原地第三次重試：
- 同一個修法失敗兩次（即使每次微調參數）。
- **打地鼠**：修 A 壞 B，修 B 又壞 A。代表抽象層選錯了。
- diff 越改越大但驗收條件的通過數沒有前進。
- 你開始想「跳過這個測試」「把驗證註解掉」「加 special case 讓它過」——這是路線錯誤的最強訊號。
- 依賴的假設堆了兩層以上都沒驗證過（「應該是 X 造成的，而 X 應該是因為 Y」）。先驗證最底層假設。

**正例**：修上傳 401 重試，改了兩次 interceptor 都會在 refresh 併發時爆 → 停手，回頭質疑「是不是不該在 interceptor 層做」，改成 single-flight refresh（換抽象層）。
**反例（誤判為方向錯）**：測試失敗但錯誤訊息每次不同且都指向新的、更深的具體原因——這是**在前進**，繼續。判斷依據：每輪是否獲得新資訊。

## §5 品質底線怎麼驗（quality floor）

任何程式碼改動，宣稱完成前依序過這關：
1. **測試**：跑受影響範圍的測試（`pytest tests/ -k <相關>`；前端 `npm test` 若有）。新邏輯要有新測試。
2. **實跑**：有 runtime 行為的改動用 `/verify` 或手動走一遍該流程，觀察實際行為（不是只看測試綠）。
3. **Diff 自審**（弱模型也要做的機械 checklist）：
   - [ ] 有沒有殘留的 debug print / 註解掉的程式碼 / TODO？
   - [ ] 有沒有硬編碼的密鑰、URL、magic number？
   - [ ] 錯誤路徑處理了嗎（不只 happy path）？
   - [ ] 改動是否比需求大（順手重構要拆開）？
4. **高風險加驗**（diff 碰到 `auth/`、`subscriptions.py`、`oauth.py`、webhook、IAM、S3 presign、加解密）：
   - 跑 `/security-review`，且按 model-dispatch.md §7 派第二意見 agent。
   - 檢查：輸入是否白名單驗證、是否可能 enumeration、簽名/HMAC 是否被弱化、憑證是否進 log。

**正例**：改了 `newebpay_service.py` 的 padding 邏輯 → 單測 + sandbox 實測一筆 + `/security-review` + 第二意見確認沒弱化 AES 處理。
**反例**：「測試都過了所以完成」——但這個 bug 本來就沒有測試覆蓋，測試過≠修好，要實跑復現場景。

## §6 Memory / 文件引用前的驗證

**判準**：memory 或文件提到的以下內容，引用前先驗證（一條指令的成本）：
- branch → `git branch -a | grep <name>`；查不到再 `git log --oneline --all | grep <關鍵字>` 看是否已合併。
- 檔案/目錄 → `ls` 或 Read 開頭幾行。
- AWS 資源 / 設定值 → 能用一條 CLI 查就查（`aws ec2 describe-instances --instance-ids ...`）。
- 驗證失敗時：把該 memory 視為歷史，**更新或刪除該 memory 檔**（這是義務，不是可選）。

**正例**：memory 說「pro-priority-queue 在 feat 分支未 push」→ `git branch -a | grep priority` 查無 → `git log` 發現 PR #166 已合併 → 更新 memory 為已完成，再繼續原任務。
**反例**：直接 `git checkout feat/pro-priority-queue` 失敗後，花 20 分鐘想「分支怎麼不見了」——第一步就該驗證。

## §7 誠實條款（harness 的極限）

拆解、驗證、多樣本評審能補**執行品質**；補不了**模糊題與品味**。遇到以下情況不要硬做：
- 產品方向、UX 品味、文案語氣、架構的長期取捨 → 產出 2-3 個具體選項 + trade-off 表，交使用者拍板（§3）。
- 需要外部事實但查不到 → 明說「查不到，以下是基於 X 假設」，禁止編造（版本號、API 行為、價格都算）。
- 自我評估「這個判斷我沒把握」→ 說出來 + 升級模型或要第二意見，這不是失敗，掩蓋才是。

## §8 Session 收尾（有未完成工作時的落檔義務）

**判準**：session 要結束（或使用者說先到這裡）而工作停在完成階梯（§2）第 ⑧ 階之前 → 結束前必須寫/更新 memory，內容**必含三件事**：
1. 停在階梯第幾階、下一步是什麼。
2. 工作所在位置（分支名 / 檔案路徑 / PR 編號）。
3. **一條可驗證現況的指令**（下個 session 用它判斷這則 memory 是否還新鮮）。

**正例**：「autosave 停在第⑥階：PR #206 已開未 merge。驗證：`gh pr view 206 --json state`。」
**反例**：memory 只寫「autosave 已實作」——沒有位置、沒有驗證指令，三週後這句話變成無法判斷真偽的漂移源（2026-07-04 盤點一次抓到六筆這種）。
