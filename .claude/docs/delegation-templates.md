# 派工 Prompt 模板

> 用法：複製對應模板 → 填 `{...}` 空格 → 刪掉不適用的行 → 發給 Agent tool。
> 每份模板已內建「三件套」（目標與動機 / 驗收條件 / 回報格式）與回報合約（model-dispatch.md §4-5）。
> 「建議 agent/model」是預設值，升降級照 model-dispatch.md §6。
> last-updated: 2026-07-04（Fable 5 建立）

---

## 1. 搜尋（找程式碼 / 找設定 / 掃 repo）

建議：`subagent_type: Explore`，model 預設繼承（不用指定）。

```
【目標】在這個 repo 找出 {要找什麼，例如：所有讀取 QUOTA_TIERS 的位置}。
【動機】{一句話，例如：要改配額邏輯，需要先知道影響面}。
【範圍】從 {目錄，例如 src/} 開始；搜索廣度 {medium|very thorough}。
  已知線索：{關鍵字 / 檔名 / 可能位置，沒有就寫「無」}。
【驗收條件】
- 每個結果附 檔案:行號 + 一行說明它在做什麼。
- 找完後明說「已窮盡」或「還有 X 沒查（原因）」。
- 一個都找不到也是合法答案，明說查過哪些 pattern。
【回報格式】條列，每條 `路徑:行號 — 說明`。總結一段（≤5 行）。
禁止貼大段原始碼；超過 30 行的內容落檔到 scratchpad 回傳路徑。
```

## 2. 實作（新功能 / 修 bug，規格已明確）

建議：`subagent_type: general-purpose`，`model: sonnet`（機械性套用已解模式用 `haiku`）。

```
【目標】{做什麼，例如：在 tags router 加 DELETE /tags/{id} endpoint}。
【動機】{為什麼 / 上游脈絡一句話}。
【規格】
- {行為描述，含輸入輸出}
- {邊界條件：權限？不存在時回什麼？}
- 遵循該檔既有慣例（三層架構：router 薄、邏輯進 service、DB 進 repository）。
【驗收條件】
- {測試指令，例如：pytest tests/test_tags.py 全過}，新邏輯有新測試。
- 不改動規格以外的檔案；發現必須改別的檔先回報再動。
- diff 自審過 judgment-rubrics.md §5 第 3 條 checklist。
【回報格式】
- 完成階梯位置（judgment-rubrics.md §2，例如「第③階：測試過」）。
- 改動檔案清單（路徑 + 各一行說明）+ 測試輸出摘要（≤10 行）。
- 卡住兩次就停，回報失敗軌跡（做了什麼、錯誤原文、你的假設），不要硬撐。
```

## 3. 重構（行為不變的結構調整）

建議：`subagent_type: general-purpose`，`model: sonnet`；跨模組的設計先派 `Plan` 出計畫。

```
【目標】把 {對象} 重構為 {目標結構}，行為完全不變。
【動機】{一句話}。
【護欄】
- 先跑 {測試指令} 記錄基線（幾個過幾個 skip），重構後結果必須一致。
- 一次一個語義步驟；禁止「順手」改行為、改命名風格、加功能。
- 公開介面（route path、函數簽名、回傳格式）不動；要動先回報。
【驗收條件】
- 基線測試結果一致（貼前後對比）。
- {若適用}行數/複雜度目標：{例如 router 降到 X 行以下}。
【回報格式】改動檔案清單、測試前後對比、有無發現但未處理的問題（列出來，不要修）。
```

## 4. 研究（技術選型 / 外部文件 / 網頁調查）

建議：`subagent_type: general-purpose`，`model: sonnet`；重大選型升 `opus` 或用 deep-research skill。

```
【目標】回答：{具體問題，例如：faster-whisper v3 的 batched pipeline 是否支援 word timestamps}。
【動機】{要做什麼決定}。
【驗收條件】
- 每個關鍵論斷附來源（官方文件/原始碼連結優先，部落格次之，標註日期）。
- 區分「文件明說」vs「你的推論」，推論要標註。
- 查不到就寫查不到 + 查過哪，禁止用訓練記憶充當現況（版本、價格、API 行為必須有來源）。
【回報格式】
- 第一行直接回答問題（是/否/視情況）。
- 證據條列（來源 + 一行摘要）。完整筆記落檔 scratchpad，回傳路徑。
- 最後一段：對「{要做的決定}」的建議與風險。
```

## 5. 審查（code review / 驗收別人的產出）

建議：fresh-context `general-purpose`；一般 diff `model: sonnet`，auth/payment/IAM 相關升 `opus`。
（審自己 session 的 working diff 優先直接用 `/code-review` skill。）

```
【目標】審查 {對象：PR #N / 分支 / 檔案清單}，判斷是否符合下列驗收條件。
【脈絡】這個改動的目的是 {一句話}。你沒有參與實作，用挑戰的眼光讀。
【驗收條件原文】（逐條核對，這是你的 checklist）
- {條件 1}
- {條件 2}
【審查重點】正確性 bug > 安全（judgment-rubrics.md §5 第 4 條清單）> 邊界條件 > 可讀性。
不要求風格重寫；只報你能說出具體失敗場景的問題。
【回報格式】
- 每個 finding：`檔案:行號 — 問題一句話 — 具體失敗場景（什麼輸入會炸）`。
- 按嚴重度排序。沒問題就明說「逐條核對過，無 finding」+ 核對紀錄。
- 結論：PASS / FAIL（對照驗收條件，逐條標 ✓/✗）。
```
